# 01. Product Planning

## 1) 프로젝트 한 줄 소개

- 프로젝트명: `mini_redis`
- 한 줄 설명: 해시 테이블 기반 인메모리 키-값 저장소를 직접 구현하고 API로 제공합니다.
- 개발 기간: 1주
- 팀 구성: 백엔드 중심 소규모 팀

## 2) 문제 정의

- 문제 1: 외부 API를 반복 호출할 때 응답 지연이 누적됩니다.
- 문제 2: 캐시 서버 동작 원리를 모른 채 라이브러리만 쓰기 쉽습니다.
- 기존 방식의 한계: Redis를 쓰더라도 내부 동작과 만료 정책을 설명하기 어렵습니다.

## 3) 타겟 사용자

- 핵심 사용자: 서버 개발 학습자/프로젝트 팀원
- 사용 상황: API 지연을 줄이고 캐시 전략을 실험할 때
- 사용 빈도/맥락: 로컬 개발/데모 환경에서 반복 조회 요청 시

## 4) 목표

- 제품 목표: `set/get/del/expire/ttl/invalidate`를 제공하는 Mini Redis API 제공
- 사용자 경험 목표: 명확한 API 계약과 예측 가능한 에러 응답 제공
- 기술 목표: TTL 만료 정책, 동시성 안전성, 성능 비교 지표 확보

## 5) MVP 범위

| 기능 | 설명 | 우선순위 | 포함 여부 |
| --- | --- | --- | --- |
| `set/get/del` | 기본 저장/조회/삭제 | High | Yes |
| `expire/ttl` | 키 만료 설정과 남은 TTL 조회 | High | Yes |
| `invalidate` | prefix 기반 대량 무효화 | Medium | Yes |
| 성능 비교 API | no-cache vs with-cache 비교 | High | Yes |
| 단위/통합 테스트 | 핵심 성공/실패/만료 검증 | High | Yes |

## 6) 비MVP 범위

- RESP(TCP) 프로토콜 완전 호환
- 영속성(AOF/RDB) 구현
- 분산 캐시/클러스터링

## 7) 핵심 사용자 흐름

### Flow A. 캐시 저장/조회

1. 사용자는 `POST /redis/set`으로 값을 저장합니다.
2. 시스템은 메모리 해시에 값을 기록합니다.
3. 사용자는 `GET /redis/get`으로 값을 조회합니다.
4. 시스템은 만료 여부를 검사한 뒤 값을 반환합니다.

### Flow B. 성능 비교

1. 사용자는 `GET /demo/no-cache`를 호출합니다.
2. 시스템은 느린 데이터 소스를 매번 조회합니다.
3. 사용자는 `GET /demo/with-cache`를 반복 호출합니다.
4. 시스템은 첫 요청 후 캐시 히트로 응답 지연을 줄입니다.

## 8) 성공 기준

- 데모 시나리오 1개 이상 완주 가능
- 핵심 API가 문서와 동일하게 동작
- TTL 만료/에러 케이스 처리 확인
- no-cache 대비 with-cache 개선 수치 확보
